Key findings in brief:

1. **Rollback journal write sequence**: SQLite saves original page content to the journal before modifying the database. The journal header starts with page count = 0 (signaling "incomplete") and only gets updated to the real count after all page data is flushed to disk - making crash recovery safe regardless of when power fails.

2. **Lock escalation**: SHARED (read) -> RESERVED (write intent, reads still allowed) -> PENDING (new reads blocked, existing reads drain) -> EXCLUSIVE (sole access, write to database). The PENDING lock is the mechanism that prevents writer starvation from a continuous stream of readers.

3. **Reader blocking**: Readers are not blocked until PENDING is acquired. Existing SHARED lock holders can finish reading. No new readers are admitted once PENDING is set. Writers wait for the last existing reader to release before escalating to EXCLUSIVE.

4. **Mode variants**: DELETE/TRUNCATE/PERSIST are all ACID-safe and differ only in what signals "committed" on disk (file absent / file empty / header zeroed). MEMORY is unsafe - journal lives in RAM only, crashes cause corruption.
